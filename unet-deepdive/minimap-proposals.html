U-Net mini-map — pick a style

The mini-map should echo scene 0's bar diagram so the viewer recognises it instantly. Below are four candidates, all rendered with the same example highlight so you can compare like-for-like. Each is what would actually sit inside a "you are here" sidebar (≈ 280 px wide).

Highlight:
proposal A+  ·  the picked one

Faithful miniature with proper U shape + landmark labels

Same 19 tensor bars and same colour-coded arrows as proposal A, but: (1) the bottleneck row is now centred horizontally and the encoder / decoder rows stair-step inward, so the silhouette is a recognisable U; and (2) seven landmark labels (input, enc1, enc2, bottleneck, dec2, dec1, out) sit above the relevant bars, while the conv1 / pool / up / concat intermediates stay unlabeled to keep the silhouette clean.

~280 px wide
bottleneck — enc3.conv2

Strengths

  • Clear U silhouette — bottleneck centred, rows indent inward.
  • Landmark labels make the highlight legible at a glance.
  • Same bars, arrows, split-colour concat as scene 0.

Trade-offs

  • Slightly taller than proposal A (label band on top adds ~14 px).
  • Still 19 bars — the smallest are barely 3 px wide.
proposal A  ·  original (no labels, left-anchored bottleneck)

Faithful miniature of scene 0

Every one of scene 0's 19 tensor bars, every colored arrow, no labels, no legend. Highlights = bright orange ring + soft glow + dimmed background. Most information; closest match to scene 0's silhouette.

~280 px wide
bottleneck — enc3.conv2

Strengths

  • Maximum visual continuity with scene 0 — same bar widths, heights, arrows.
  • Shows that each block is two convs (two adjacent bars).
  • Concat tensors are split-coloured exactly like scene 0.

Trade-offs

  • 19 elements at this size = visually busy.
  • Some bars (input, output, 1×1) are very small.
proposal B

Block-level miniature

One bar per block (the conv1+conv2 pair collapses into a single bar with the block's final shape). 5 main blocks + ups + concats + 1×1 head + output. ~10 visual tensors, larger individually. Same colored arrows.

~280 px wide
bottleneck — enc3

Strengths

  • Cleaner; each bar is bigger and easier to see.
  • Still preserves the U-shape silhouette and bar-geometry rule (height ∝ spatial, width ∝ channels).
  • Matches the "block-level" mental model the rest of the deepdive uses (`enc1`, `enc2`, …).

Trade-offs

  • Hides the "two convs per block" fact (scene 4 / scene 10 explain it explicitly).
  • One bar = one block = the same simplification scene 0 was designed to fight.
proposal C

Block-level with labels

Same as B but each bar gets a tiny label (`enc1`, `enc2`, …). Easier to anchor when the highlight changes.

~280 px wide
bottleneck — enc3

Strengths

  • Labels make the highlight legible without the caption underneath.
  • Reads as "shrunken textbook diagram".

Trade-offs

  • Tiny text at this size. May force the SVG to be a bit wider.
proposal D

19-bar silhouette, no arrows

All 19 bars from scene 0, but the operation arrows are stripped out — only the bar silhouette remains. Highlights still work via the orange ring + glow. The most "thumbnail" of the four; reads almost like a logo of the architecture.

~280 px wide
bottleneck — enc3.conv2

Strengths

  • Cleanest silhouette of scene 0.
  • Less visual noise — the orange highlight pops more.

Trade-offs

  • No operations visible — viewer can't see "this is where pool happens".
  • Skip arcs gone, so the U-Net's signature shortcut is invisible.

My recommendation: proposal C (block-level with labels). It carries enough visual continuity with scene 0 (same Ronneberger bars, same colors, same U-shape) without the noise of all 19 elements at thumbnail scale, and the labels make the highlight unambiguous at a glance. Proposal A is the most faithful but probably too dense at 280 px wide.
